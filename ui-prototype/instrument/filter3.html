Multi-Context Comparison View

Prototype exploring different approaches for implementing Filter (selection) and View Options (evaluation) controls. The separation ensures users explicitly select data scope while system-derived states control presentation.

Filter (Selection)

  • Selecting specific attributes
  • Selecting attribute groups
  • Selecting relationships
Rule: If the user explicitly chooses what to include, it belongs in Filter

View Options (Evaluation)

  • Attributes with errors
  • Mandatory attributes
  • Attributes pending approval
  • Differences: Show All | Highlight | Only Diff (mutually exclusive)
Rule: If the system determines it, it belongs in View Options

Design Rationale:
  • Color coding: Blue for Filter (selection intent), Purple for View Options (evaluation intent)
  • Spatial separation: Two distinct panels reinforce conceptual separation
  • Safety badges: "Always Visible" badges on error/mandatory options clarify safety invariants
  • Apply vs. Instant: Filter requires explicit "Apply" (changes scope), View Options can be instant toggles (presentation only)
Query Builder Approach (matching existing 1D table pattern):
  • Same pattern: "Where [Field] [Operator] [Value]" - consistent with existing table filters
  • Pick a field: Dropdown to select what to filter on (Attribute Group, Name, Has Difference, etc.)
  • Context-aware operators: Text fields get contains/equals, booleans get Yes/No toggle
  • Stackable conditions: Add multiple conditions with "And" - all must match
  • Cancel/Apply: Review conditions before applying, with clear cancel option
  • Results status: Shows "X of Y attributes" with clear all option
  • Scales infinitely: No list of items - just build the query you need
Filter Fields for 2D Table:
  • Attribute Group: Filter by group (Basic Info, Pricing, Specs, etc.)
  • Attribute Name: Text search on attribute names
  • Attribute Type: Show only Attributes or only Relationships
  • Has Difference: Show only rows with differences between contexts
  • Has Error: Show only rows with validation errors
  • Is Pending: Show only rows pending review
  • Value in Source A/B: Search within specific context values

Design Recommendations

Summary of approaches and when to use each pattern.

When to Use Each Pattern

Pattern Best For Pros Cons
Dual Side Panels Complex filtering with many attribute groups and deep hierarchies Maximum clarity, supports complex trees, clear separation Takes significant screen space, may feel heavy for simple cases
Toolbar Dropdowns Simpler scenarios with fewer options, quick access Compact, familiar pattern, doesn't steal focus Nested content awkward, limited space for descriptions
Chips Bar Making current state always visible, quick removal Transparent state, easy to clear individual items Gets tall with many active filters, still needs panels to add
Split Inline Panel Moderate complexity, frequent adjustments to both Quick toggle access, both visible, collapsible Limited to flat lists, not for deep hierarchies
Key Design Principles Applied:
  • Visual Differentiation: Blue for Filter (user selection), Purple for View Options (system evaluation)
  • Safety Invariants: Errors and mandatory data badges show these are always visible
  • Clear Labels: "Select what to include" vs "Control how data is displayed"
  • Appropriate Scope: Filter changes require "Apply" (scope change), View Options can be instant (presentation only)
