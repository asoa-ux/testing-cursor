MDM Form Layout Analysis

Evaluating approaches for product detail form field widths

1 The Problem

Users have reported that the product detail form "does not use space efficiently."

The current layout has:

  • Single-column structure with inline labels (label left, field right)
  • Column-based width system (96px columns + 24px gutter) — field types map to 2, 3, 4, or 5 columns
  • Short fields waste space — SKUs, dates, numbers use only a fraction of available width
User Feedback

"We do not use the space efficiently" — Users want better use of horizontal space and less visual inconsistency.

2 Proposed Change: Simplify to 3–4 Responsive Sizes

Map the existing column-based widths (2, 3, 4, 5 columns) to named sizes that work responsively. The number of sizes depends on the layout approach:

Option B (Multi-column Grid): 4 sizes

Each size maps to a column span in a 4-column grid:

3 Four Layout Approaches

We can apply the size system in different ways:

A. Original (Current) Status Quo

Single column, inline labels, column-based widths (96px columns + 24px gutter, mapped to 2-5 columns per field type).

Pros
  • Familiar to users
  • Easy vertical scanning
  • Existing system, no work needed
Cons
  • Doesn't address space complaint
  • Wasted horizontal space
  • Fixed pixel widths, not responsive

B. Multi-column Grid Alternative

4-column grid with top labels. Sizes map to column spans (S=1, M=2, L=3, F=4). Responsive: collapses to 2 columns, then 1 column on mobile.

Pros
  • Multiple small/medium fields per row
  • Better horizontal space usage
  • Good for reviewing many fields at once
Cons
  • Top labels add height — full-width fields take MORE space than inline labels
  • Scrolling savings are marginal (depends on field mix)
  • Z-pattern scanning (slower to find specific field)
  • Denser, may feel overwhelming
  • More complex to implement

C. Single Column + Mapped Sizes Recommended

Keep single-column layout and inline labels. Since fields are alone on each row, effectively 3 widths: Small (25%), Medium (50%), Large/Full (100%). Below 600px, labels move to top and all fields become full width.

Pros
  • Addresses visual inconsistency (proportional sizing)
  • Inline labels are more space-efficient than top labels
  • Easy vertical scanning to find fields
  • Familiar layout preserved
  • Less disruptive, lower risk
Cons
  • Horizontal space to the right of small fields remains empty
  • Same number of rows as before

D. Two Column Layout Split Panel

Two Layout C panels side by side, each displaying different attribute groups for the same record. Useful for comparing or organizing attributes into logical categories (e.g., marketing vs. logistics). Includes shared toolbar and dual sidebar configuration.

Pros
  • Displays more attributes simultaneously
  • Logical grouping (e.g., identity/marketing in left, pricing/inventory in right)
  • Good for power users who need to see related data together
  • Reduces context switching between tabs/sections
Cons
  • Requires wider viewport — not suitable for narrow screens
  • 240px labels may be too wide for narrower columns
  • Higher cognitive load with two forms visible
  • With two sidebars, horizontal space becomes constrained
  • No responsive breakpoint to stack columns vertically
Why Option C?

The "space efficiency" complaint is likely about visual inconsistency rather than actual space usage. Option C addresses this with proportional sizing while preserving the familiar scanning pattern. Option B's multi-column layout adds height via top labels, so scrolling savings are marginal — and the Z-pattern scanning makes finding specific fields slower. Option D is best suited for power users on wide displays who benefit from seeing multiple attribute categories simultaneously.

Future consideration: Use the whitespace purposefully

Instead of eliminating whitespace, consider using it for contextual information: field hints, validation feedback, data quality indicators, last-modified metadata, or quick actions. This reframes the "wasted" space as valuable real estate that improves data entry — without changing the layout.
