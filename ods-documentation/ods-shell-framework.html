The Application Shell

A shared layout grammar for every product in the Stibo Systems suite. This document defines the five structural layers, their responsibilities, their interaction rules, and the decision criteria designers use when building new — or rearchitecting existing — applications.

Design principles

These principles govern decisions at every layer. When two guidelines conflict, apply the one ranked higher.

01Scope stays in its layer

Each layer has a defined jurisdiction. Navigation belongs in A and B. Deep data browsing belongs in C. Working on content belongs in D. Contextual inspection belongs in E. Never use a layer for something outside its scope.

02Global before local

Layer A actions affect the whole suite. Layer B actions affect the current product. Layer C actions affect the current section. Layer D actions affect the current item. Never surface a global action below Layer A.

03Selection cascades downward

Selecting in Layer B updates Layer C (if present) and Layer D. Selecting in Layer C updates Layer D. This cascade is one-way and must not be reversed. Layer D actions must never change the Layer B navigation state.

04Optional layers are triggered, not defaulted

Layer C appears when the current context contains a deep hierarchy to browse. Layer E appears when the user explicitly requests contextual information or AI agent interaction. Neither panel should appear on page load unless the context demands it.

05Consistency over customisation

Product teams may configure Layer B navigation items, Layer C data sources, and Layer D content. They may not relocate components between layers, add new persistent layers, or break the left-to-right cascade order.

Layer A — Top Bar

The Top Bar is the one persistent surface shared across all products. It orients the user within the suite and provides access to suite-level functionality.

ℹLayer A is owned by the ODS team. Product teams do not modify its structure or component order. They may supply a customer logo and configure which apps appear in the app switcher.

Behaviour

The Top Bar is always visible and never scrolls with the page content. It sits above all other layers at a fixed height. It never contains product-specific navigation links or data-level actions.

⚠Do not place product navigation in Layer A. If an item feels like it belongs between the app switcher and search, it is almost certainly a Layer B item.

Layer B — Side Panel

The Side Panel provides navigation within a single product. It is shallow by rule — users should never need to descend more than two levels to reach any navigable section.

Anatomy — top to bottom

SlotComponentRequiredNotes
recentRecently visited itemsOptionalProduct team decides whether to include this; maximum 5 items shown
nav-itemsProduct navigationYesAccordion or popover based on depth rules below
settingsSettingsIf applicableOnly present when the product has configuration screens; always at the bottom of nav, above the user block
user-infoUser blockYesAvatar, display name, email; always pinned to bottom
company-logoCompany / product logoYesAlways pinned to bottom, below user block

Depth rule

Layer B navigation must never carry deep data hierarchies. The maximum depth is two levels (a parent item and its direct children). Any structure deeper than two levels belongs in Layer C.

✕Never replicate MDM data trees in Layer B. A product hierarchy with hundreds of nodes is a Layer C concern, not a Layer B concern.

Collapsed state

When the user collapses Layer B via the button in Layer A, the panel is fully hidden — not reduced to an icon rail. The Layer A collapse button changes to an expand icon. This maximises the working area in Layer D and is the preferred state for data-intensive tasks.

Layer C — Explorer

The Explorer is a contextual data panel. It is not global navigation. It appears when the current section of the product exposes a deep hierarchy that the user needs to browse — for example, a product category tree or an organisational structure.

When to include Layer C

Ask the following question: "Does the user need to navigate a structured hierarchy with potentially unlimited depth before they can work on an item?" If yes, Layer C is appropriate. If the user only needs to pick from a flat list or a two-level structure, Layer B is sufficient.

ℹLayer C is driven by the Layer B selection. Switching sections in Layer B must update — or hide — Layer C accordingly.

Responsibilities

  • ✓ Hierarchical browsing of MDM data structures
  • ✓ Node selection that updates the Layer D content
  • ✓ May be as deep as the data requires — no depth limit applies here
  • ✕ Must not contain suite navigation or product section links
  • ✕ Must not duplicate Layer B navigation items

Layer B + C co-existence

Layer B always remains fully expanded when Layer C is open. There is no automatic icon-rail state. If a user wants to maximise horizontal space, they collapse Layer B manually using the collapse button in Layer A — this is a user action, not a product-driven behaviour. Product teams must not programmatically hide or collapse Layer B when Layer C appears.

Layer D — Content Area

Layer D is the primary working surface. It is always present. Its content is entirely determined by the current selection state in Layer B and, when applicable, Layer C.

General rules

Layer D must never contain navigation elements that duplicate what is available in Layers B or C. All persistent navigation belongs to the left. Layer D is a workspace, not a menu.

Layer D supports two page type patterns, defined below in Page Types. Product teams may create additional page types for their domain but must follow the shared structural rules for breadcrumbs, titles, and action placement.

Layer E — Contextual Panel

Layer E is an on-demand panel that slides in from the right edge of the viewport. It serves two distinct modes that share the same panel surface.

ModeTriggerContent
InspectorUser clicks "View details in context" on an item in Layer DRead-only or quick-edit view of item metadata, without leaving the current Layer D page
AI AgentUser clicks the AI Agent button in Layer AFull conversational AI interface; context-aware of the current product, section, and item selection

Panel behaviour

Layer E opens as an overlay that pushes Layer D content leftward — it does not sit on top of it. Only one mode is active at a time. Switching between Inspector and AI Agent modes while the panel is open replaces the panel content without closing and reopening the panel.

The panel is dismissed by clicking the close button inside Layer E, or by pressing Esc.

ℹThe AI Agent in Layer E is context-aware. It receives the identity of the current product, the current work area, and the currently selected item. Product teams are responsible for passing this context payload when opening Layer E in AI Agent mode.

Layer B navigation interaction rules

Navigation items in Layer B follow one of two interaction patterns depending on the number of sub-items they contain. The choice is determined by a single rule.

PatternUse whenSub-item count
AccordionThe sub-items are finite and predictable1–5 items
PopoverThe sub-item list could grow, or exceeds 5 items6+ items, or open-ended
ℹWhen a popover contains more items than can be comfortably displayed, the visible items are shown first and the remainder are accessible via a "… More [X]" menu item at the end of the list. This keeps the popover scannable without truncating access.

Accordion pattern

B-01Hover state

When the user hovers a nav item that uses the accordion pattern, the item's icon is replaced by a chevron pointing down. This is the affordance that signals the item is expandable.

B-02Click behaviour

Clicking the item toggles the accordion open or closed. The chevron rotates to point upward when the accordion is open. Sub-items are revealed inline beneath the parent item. Only one accordion may be open at a time; opening a second one collapses the first.

Popover pattern

B-03Chevron indicator

Items using the popover pattern display a static chevron pointing right on the trailing edge of the item at all times — not just on hover. This distinguishes them from accordion items.

B-04Click behaviour

The entire item row is clickable and opens the popover. The popover lists all sub-sections. At the end of the popover list, a "View all [X]" menu item links to a full listing page for that section. Popovers are dismissed on outside click or Esc.

⚠Do not mix accordion and popover patterns for sibling items at the same navigation level within a single product. Choose one pattern per depth level and apply it consistently.

Page types in Layer D

Two canonical page structures cover the majority of use cases. Every screen in a product should map to one of these — or explicitly document why it diverges.

Bulk page — lists, tables, galleries

Used when the user is browsing or operating on multiple items at once.

Bulk page structure

Breadcrumb navigation context
Page title … actions button → far right
Search field + Sort + Filter
Actions · … overflow · View toggle (far right)
Table or gallery content fills remaining height
Sticky footer: total item count · X of Y selected (shown when items are selected) sticky · bottom
D-01Actions placement

All actions — both general page actions and bulk actions for selected items — live in the same action area on the right side of the search/filter bar. Actions are shown as buttons; when there are more actions than space allows, they overflow into a "…" icon button at the far right of that row. There are no separate bulk-action controls in the footer — the footer shows selection count only.

D-02View mode toggle

If a bulk page supports both table and gallery views, the toggle sits at the far right of the search/filter/actions bar — it is the rightmost element in that row. Table view is the default. The chosen mode persists per user per work area.

D-03Sticky footer

The footer always shows the total number of items in the current filtered set. When one or more items are selected, it additionally shows "X of Y selected". The footer is informational only — actions remain in the toolbar row above the list. The footer must stay visible while scrolling.

Details page — single item

Used when the user is reading or editing a specific record.

Details page structure

Breadcrumb shows full path to item
Item title … actions button → far right
Secondary metadata (revision, last editor, status badge…) below title
Tab bar (Details / References / Activity / …) sticky below header
Item details
Form fields grouped by named sections · collapsible
⚠To be defined: How widgets are handled within the Details page and the Bulk (table/gallery) page. This section will be updated once the widget pattern is agreed upon.
D-04Tabs

The tab bar is sticky below the item header. The first tab is always the primary data view ("Details"). Additional tabs (References, Activity, etc.) are product-specific. Do not add navigation-level links as tabs — tabs switch the content view of the same item, not the item itself.

D-05Field grouping

Form fields within the Details tab are organised into collapsible named sections. Related fields belong together in the same section. Section names are title case and describe the content, not the structure (e.g. "Product information", not "Section 1").

Responsive — Tablet behaviour

The shell is designed for desktop-first usage. Tablet support (≥ 768px viewport width) is required. Phone support is out of scope for this version.

LayerDesktop (≥ 1280px)Tablet (768px – 1279px)
Layer AFull — all slots visibleCustomer logo hidden; env-context collapsed into a single icon
Layer BFull width, visible by defaultHidden by default; tap expand button in Layer A to open as a full overlay
Layer CInline panel beside Layer BOpens as an overlay drawer from the left edge; tap outside to dismiss
Layer DFills remaining widthFull viewport width when B and C are hidden
Layer EPushes Layer D content leftwardOpens as a full-width bottom sheet instead of a side panel
⚠On tablet, avoid designs that require Layer B, C, and E to be simultaneously visible. The available horizontal space is insufficient for a comfortable experience with all panels open at once.
