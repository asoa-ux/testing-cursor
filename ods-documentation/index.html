One Design System

ODS Documentation

A unified design language for the entire product portfolio. ODS provides the shared grammar — foundations, components, patterns, and guidance — that enables teams to build cohesive experiences across every product.

What is ODS?

The One Design System unifies our fragmented design landscape — three existing design systems plus a portfolio of products that follow no system at all — into a single source of truth. ODS creates a centralized hub where all products share common patterns while maintaining their distinct purposes.

Why ODS exists

Today, our product portfolio is fragmented: three separate design systems serve some products, while many others have evolved without any systematic design guidance. This creates inconsistent user experiences and duplicated effort across teams.

  • Consistency — Users moving between products experience a familiar interface language
  • Efficiency — Design and engineering teams work from shared components and patterns instead of reinventing solutions
  • Quality — Centralized documentation ensures best practices are applied everywhere
  • Scalability — New products inherit a mature design foundation from day one
  • Migration path — Products without a system gain clear guidance for adopting ODS

The five-layer framework

At the core of ODS is the Application Shell — a shared layout structure that every product inherits. This framework divides the interface into five functional layers:

  • Layer A — Top Bar: Suite-level navigation, global search, AI agent entry
  • Layer B — Side Panel: Product-specific navigation (shallow hierarchy)
  • Layer C — Explorer: Deep data hierarchy browsing (optional)
  • Layer D — Content Area: The primary workspace
  • Layer E — Contextual Panel: Inspector or AI agent interaction (optional)

Contributing

ODS is a living system. Product teams can propose changes, additions, or clarifications.

How to propose changes

  • Minor clarifications — Comment directly in the Figma library or raise in the #ods-design Slack channel
  • New components or patterns — Submit an RFC (Request for Comments) using the template in Confluence
  • Breaking changes — Requires review by the ODS core team and a migration plan

Governance

The ODS core team reviews proposals weekly. Decisions are documented in the changelog. Product teams are notified of changes that affect their implementations.

Glossary

Key terms used throughout this documentation.

  • Application Shell — The shared layout structure (Layers A–E) that all products inherit
  • Bulk page — A page type for browsing and acting on multiple items (lists, tables, galleries)
  • Details page — A page type for viewing and editing a single item
  • Explorer — Layer C; a panel for navigating deep data hierarchies
  • Inspector — A mode of Layer E that shows contextual information about a selection
  • Layer — One of five functional regions in the application shell
  • Suite — The collection of all products accessible through the Top Bar app switcher
  • Token — A named design value (colour, spacing, typography) that ensures consistency
  • Wayfinding — How users navigate and orient themselves within and between pages
  • Workspace — The environment/tenant context a user is working within

1.1 Design tokens

Design tokens are the smallest building blocks of the design system. They capture design decisions as named values that can be used in both design tools and code.

Token categories

  • Colour — Semantic colour tokens for text, backgrounds, borders, and status
  • Typography — Font family, size, weight, and line height scales
  • Spacing — Consistent spacing values based on an 8px grid
  • Radius — Border radius values for different component sizes
  • Shadow — Elevation levels for layering and depth
  • Motion — Duration and easing curves for animations
✓ Token documentation exists in the Figma library. Reference the ODS Tokens file for current values and usage guidelines.

1.2 Typography scale

ODS uses the DM Sans family as the primary typeface, with DM Serif Display for display headings and DM Mono for code and technical content.

Type scale

  • display-lg — 48px / DM Serif Display / Page headings
  • display-md — 28px / DM Serif Display / Section titles
  • heading-lg — 20px / DM Sans Semibold / Card titles
  • heading-md — 15px / DM Sans Semibold / Subheadings
  • body — 15px / DM Sans Regular / Body text
  • body-sm — 13px / DM Sans Regular / Secondary text
  • caption — 11px / DM Sans Medium / Labels, metadata
  • code — 12px / DM Mono / Code snippets
♿ Accessibility: Body text must be at least 14px. Line height should be at least 1.5× font size for readability. See §8 Accessibility →

1.4 Spacing & grid

Spacing values are based on an 8px base unit, creating a consistent rhythm across all interfaces. The 12-column grid provides the foundational layout structure for Layer D content.

⚠ To be documented: Complete spacing scale, 12-column grid specifications, and layout guidelines.

Spacing scale

  • space-1 — 4px (half unit)
  • space-2 — 8px (base unit)
  • space-3 — 12px
  • space-4 — 16px
  • space-6 — 24px
  • space-8 — 32px
  • space-12 — 48px
  • space-16 — 64px

12-column grid

Layer D uses a 12-column grid for content layout. The grid adapts to the available width based on which layers are visible.

  • Column count: 12
  • Gutter width: 24px
  • Margin: 32px (desktop), 24px (tablet)
  • Common spans: 12 (full), 6 (half), 4 (third), 3 (quarter)

Breakpoints

  • desktop — ≥ 1280px
  • tablet — 768px – 1279px
  • mobile — < 768px (out of scope for v1)

1.7 Iconography

ODS uses a custom icon library designed for clarity at small sizes.

Icon sizes

Usage rules

Quick reference: Layer responsibilities

  • Layer A (Top Bar) — Product switching, workspace selection, global search, notifications, AI agent entry. Always present.
  • Layer B (Side Panel) — Shallow product navigation (max 2 levels). Determines what appears in Layer C. Always present.
  • Layer C (Explorer) — Deep structural browsing of MDM hierarchies. Node selection updates Layer D. Optional.
  • Layer D (Content Area) — Primary working surface. Always present.
  • Layer E (Contextual Panel) — Inspector mode or AI agent interaction. Optional, triggered by user.

3.7 AI Agent (Layer E mode)

The AI Agent surfaces in Layer E and receives context about the user's current state. This is one of two modes for the Contextual Panel (the other being Inspector).

⚠ Evolving: AI agent patterns are being developed in parallel with agent capabilities.

Entry points

  • Layer A button — AI agent icon in the Top Bar opens Layer E in agent mode
  • Contextual triggers — Specific actions may suggest AI assistance

Context payload

When opened, the AI agent receives:

  • Current product identifier
  • Current section (Layer B selection)
  • Current item (Layer D selection, if applicable)
  • User role and permissions

Interaction principles

  • Agent is context-aware — it knows what the user is viewing
  • Agent actions require explicit user confirmation
  • Agent can be dismissed without losing work context
  • Conversation persists within a session
  • Switching between Inspector and Agent mode preserves each mode's state

Planned content

  • Suggested actions UI pattern
  • Confirmation dialogs for agent-initiated actions
  • Error and fallback states
  • Loading states during agent processing

4.1 Bulk page — lists, tables, galleries

Used when the user is browsing or operating on multiple items at once.

Structure (top to bottom)

  • Breadcrumb — Navigation context showing the path to current view
  • Title row — Page title on the left; "..." actions button on the far right
  • Toolbar — Search field + sort + filter on the left; actions, overflow menu, view toggle on the right
  • Content area — Table or gallery, fills remaining height
  • Sticky footer — Total item count; "X of Y selected" when items are selected

Key rules

  • Actions (page-level and bulk) live in the toolbar row, not the footer
  • View mode toggle (table/gallery) is the rightmost element in the toolbar
  • Footer is informational only — no actions
→ Decision guide: Which table type?
ℹ Detailed specifications with diagrams available in the Shell Framework documentation.
♿ Accessibility: Tables must use proper <th> headers. Row selection uses checkbox pattern with aria-selected. See §8 Accessibility →

4.2 Details page — single item

Used when the user is reading or editing a specific record.

Structure (top to bottom)

  • Breadcrumb — Full path to the current item
  • Title row — Item title on the left; "..." actions button on the far right
  • Secondary metadata — Revision info, last editor, status badges
  • Tab bar — Details, References, Activity, etc. (sticky below header)
  • Content area — Form fields grouped in named, collapsible sections

Key rules

  • First tab is always "Details" (primary data view)
  • Tabs switch content view of the same item — not navigation
  • Form fields are organized into named sections, not arbitrary divisions

5.1 Hierarchy navigation

In hierarchical data structures, items may have both properties (details) and nested children. This pattern defines how users navigate between these views.

Two views per item

  • Details page — The item's properties; the canonical view of the record
  • Children page — A bulk page listing items nested below this item

Items that cannot have children only have a details page.

Entry point determines destination

  • From Layer C (Explorer) → Opens children page (user is drilling down the hierarchy)
  • From search, links, or direct URL → Opens details page (user is inspecting a specific item)

Navigating between views

  • Details → Children: Click the "X items" link in the secondary metadata row
  • Children → Details: Click the item name segment in the breadcrumb

Breadcrumb format

  • Details page: Path › Item Name
  • Children page: Path › Item Name › Items

The "Items" segment (or a context-specific term) distinguishes the children bulk page. Clicking the item name navigates to its details.

5.2 Breadcrumbs

Breadcrumbs show the user's location within the information hierarchy and provide quick navigation to parent levels.

Structure

  • Segments are separated by ›
  • All segments except the last are clickable links
  • The last segment represents the current page

Hierarchy navigation

When an item has both a details page and a children page, the breadcrumb distinguishes between them:

  • Details page: Path › Item Name
  • Children page: Path › Item Name › Items

To be defined

  • Maximum depth before truncation
  • Truncation pattern for long paths
  • Tablet adaptation
♿ Accessibility: Breadcrumbs use <nav aria-label="Breadcrumb"> with an ordered list. Current page is marked with aria-current="page". See §8 Accessibility →

8. Accessibility

Cross-cutting requirements that apply to everything in ODS. Accessibility is built into every component and pattern, not treated as an afterthought.

ℹ Throughout this documentation, look for the ♿ Accessibility callouts for component-specific guidance.
⚠ To be documented: Comprehensive accessibility guidelines are in development.

Compliance target

WCAG 2.1 Level AA is the baseline for all ODS components and patterns.

Keyboard navigation

  • Focus order — Logical tab sequence following visual layout
  • Focus visibility — Clear focus indicator on all interactive elements
  • Focus trapping — Focus stays within modals and Layer E when open
  • Keyboard shortcuts — Standard shortcuts (Escape to close, Enter to confirm)

Screen reader support

  • Semantic HTML — Proper heading hierarchy, landmarks, lists
  • ARIA patterns — Documented for each component
  • Live regions — Announcements for dynamic content changes
  • Labels — All form controls have accessible labels

Visual accessibility

  • Colour contrast — 4.5:1 for text, 3:1 for UI elements
  • Not colour alone — Information conveyed by colour also uses icons, text, or patterns
  • Text sizing — Interfaces support up to 200% text zoom
  • Target size — Interactive elements are at least 44×44px touch target

Motion

  • Reduced motion — Respect prefers-reduced-motion setting
  • No auto-play — Animations don't start without user action
  • Pause controls — Moving content can be paused
