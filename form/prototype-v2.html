Form field architecture — V2

Framework — how to read this prototype

MDM fields mix many concerns. A single “input” component is not enough; placement follows what kind of thing each signal is.

1. What makes this hard

  • Field types — Single value, multi-value, references (chips), rich text — different editing and density.
  • Sources — Local, inherited, calculated, external, “inherited + calculated” combinations, overwritten — different editability and story.
  • Levels of information — Mandatory, needs review (highlighted), help text, description text, AI active, attribute-level error, value-level error — not all attach to the same place.
  • Formats — Label left, label top, table cell, table row, whole table — different space.
  • Visibility — A problem in a cell or a “needs review” marker may be off-screen when the user scrolls; row-level cues complement cell-level markers.

2. Three separations (what each does in the UI)

Information

Understanding the field and value: help, source/provenance, description, AI story, etc.

In the table, field-level context is summarized in the corner triangle (compact). It does not by itself drive the row bar; the row bar is for validity / submission.

Validity / submission

Things that block or gate progress: mandatory, errors, needs review.

In table cells, one marker slot at a time: error > needs review > mandatory. (Form view keeps mandatory * on the label independently of needs-review styling in the field.) Row markers (red / blue bar) so issues stay visible when the cell is scrolled away.

Actions

Things the user does to the value: AI, value history, format, etc.

Shown on demand in the toolbar (form) or cell actions (table), usually on hover or focus.

Value history: the content is informational, but opening it is something the user does — so the control lives with actions, not with the label.

3. Priority roles

Information — secondary priority; keep compact (triangle, badges, labels) so the row does not feel noisy.

Validity / submission — demand attention; markers stay visible while editing; row markers help when the cell is off-screen.

Actions — tools on demand; lower visual weight than errors and mandatory until the user engages the field.
